# LLM Guide: SDK architecture and usage

This document is optimized for coding agents. It summarizes the public SDK surface, the internal composition, and the
signing/submission pipeline. Keep this file aligned with source code, not only with README examples.

## Official Resources

- **Official API Documentation**: https://docs.bulk.trade
- **API Reference**: https://docs.bulk.trade/api-reference
- **Manage Agent Wallet API**: https://docs.bulk.trade/api-reference/manageAgentWallet
- **Main Website**: https://bulk.trade

## Installation

Published package names:

- JSR: `@posaune0423/bulk-ts-sdk`
- npm: `bulk-ts-sdk`

Deno via JSR:

```bash
deno add jsr:@posaune0423/bulk-ts-sdk
```

Bun:

```bash
bun add bulk-ts-sdk
```

npm:

```bash
npm install bulk-ts-sdk
```

pnpm:

```bash
pnpm add bulk-ts-sdk
```

Runtime imports use the npm-style package name in Bun, Node.js, and the example project:

```ts
import { BulkClient } from "bulk-ts-sdk";
```

For Deno projects that install through JSR, use the import specifier generated by `deno add`, or add an import-map alias
if the consuming app wants to keep `import { BulkClient } from "bulk-ts-sdk"`.

The SDK package includes the native `bulk-keychain` bindings required for signing. Public market-data calls do not need
a private key or native signing path.

## Entry points

- Public export surface: `src/mod.ts`
  - Re-exports `BulkClient`, config types, domain types, builders, signer helpers, and SDK errors.
- Main client: `src/client.ts`
  - `BulkClient` composes HTTP, WebSocket, account, market, trade, and signing dependencies.

## Client configuration

```ts
import { BulkClient } from "bulk-ts-sdk";
import process from "node:process";

const publicClient = new BulkClient();

const mainWalletClient = new BulkClient({
  privateKey: process.env.MAIN_WALLET_PRIVATE_KEY!,
});

const agentWalletClient = new BulkClient({
  privateKey: process.env.AGENT_WALLET_PRIVATE_KEY!,
  accountPublicKey: process.env.MAIN_WALLET_PUBLIC_KEY!,
});
```

Use exactly one wallet setup:

- Main wallet only: set `MAIN_WALLET_PRIVATE_KEY` and pass it as `privateKey`.
- Agent wallet for a main wallet: set both `AGENT_WALLET_PRIVATE_KEY` and `MAIN_WALLET_PUBLIC_KEY`; pass the agent key
  as `privateKey` and the main wallet public key as `accountPublicKey`.

`BulkClientConfig` fields:

- `httpUrl`: optional HTTP API base URL. Defaults to production.
- `wsUrl`: optional WebSocket API URL. Defaults to production.
- `privateKey`: optional signing key. Required for `client.trade.*` signing methods.
- `accountPublicKey`: optional target account public key. Only needed when `privateKey` belongs to an agent wallet and
  the action should target a separate main account.
- `timeoutMs`: optional request timeout.
- `validateResponses`: currently part of the config type; do not assume it changes runtime behavior unless source code
  implements it.

`client.accountPublicKey` is derived from the signer when only `privateKey` is provided. If `accountPublicKey` is passed,
`client.accountPublicKey` is that target account public key.

## Environment variables

The SDK core does not read environment variables directly. Consumers should read env values in their own runtime and pass
them into `new BulkClient(...)`.

Ordinary TypeScript example for Bun or Node.js:

```ts
import process from "node:process";

const client = new BulkClient({
  privateKey: process.env.MAIN_WALLET_PRIVATE_KEY!,
  httpUrl: process.env.BULK_HTTP_URL,
  wsUrl: process.env.BULK_WS_URL,
});
```

Recommended variable names by context:

- Application code: choose app-specific names, then pass values into `BulkClientConfig`.
- `examples/simple-use`: `MAIN_WALLET_PRIVATE_KEY`, `BULK_HTTP_URL`, `BULK_WS_URL`, `BULK_SYMBOL`.
- Repo E2E tests: `MAIN_WALLET_PRIVATE_KEY`, `AGENT_WALLET_PRIVATE_KEY`, `MAIN_WALLET_PUBLIC_KEY`,
  `MULTISIG_PUBKEY`, `BULK_HTTP_URL`, `BULK_WS_URL`.

`BULK_HTTP_URL` and `BULK_WS_URL` are optional overrides. If omitted, the client uses the production defaults from
`src/constants.ts`.

For normal trading, do not require the main wallet public key separately; `client.accountPublicKey` is derived from
`MAIN_WALLET_PRIVATE_KEY`. For agent-wallet signing, `MAIN_WALLET_PUBLIC_KEY` is required because the signer key belongs
to the agent wallet while the target account is the main wallet.

Important naming rule:

- `privateKey` is the key that signs.
- `accountPublicKey` is the account being queried or acted on.
- The SDK config and client property are both `accountPublicKey`.

## Native signing limitation for agent wallets

Agent wallet registration/removal is supported through `client.trade.manageAgentWallet()`.

Order signing for a separate target account is guarded locally until the native `bulk-keychain` signer can sign the target
account into the signature. Current behavior:

- `new BulkClient({ privateKey: AGENT_WALLET_PRIVATE_KEY, accountPublicKey: MAIN_WALLET_PUBLIC_KEY })` constructs.
- `client.accountPublicKey` returns the main wallet public key.
- `placeLimitOrder`, `placeMarketOrder`, `cancelOrder`, `cancelAll`, and `batch` throw before submission if the target
  `accountPublicKey` differs from the native signer pubkey.
- The error message contains: `target-account signing support`.
- `manageAgentWallet` uses native `signAgentWallet(agent, remove)` and is not blocked by that guard.

Reference: https://docs.bulk.trade/api-reference/manageAgentWallet

## Core composition

`src/client.ts` creates:

- `HttpTransport` from `src/http/http_transport.ts`
- `WsClient` from `src/ws/ws_client.ts`
- `KeychainSigner` from `src/signing/keychain_signer.ts` when `privateKey` is present
- `MarketClient` from `src/http/market_client.ts`
- `AccountClient` from `src/http/account_client.ts`
- `TradeClient` from `src/http/trade_client.ts`

Sub-clients:

- `client.market`: public market data.
- `client.account`: account and multisig reads.
- `client.trade`: signed order and agent-wallet actions.
- `client.ws`: subscriptions and optional WebSocket transaction submit.

## Market client

Implementation: `src/http/market_client.ts`. These methods do not require signing.

```ts
const info = await client.market.exchangeInfo();
const ticker = await client.market.ticker("BTC-USD");
const candles = await client.market.klines({
  symbol: "BTC-USD",
  interval: "1m",
  limit: 10,
});
const book = await client.market.l2Book({
  symbol: "BTC-USD",
  nlevels: 20,
  aggregation: 1,
});
const stats = await client.market.stats({ symbol: "BTC-USD" });
const risk = await client.market.riskSurfaces("BTC-USD");
```

Endpoint mapping:

- `exchangeInfo()` -> `GET /exchangeInfo`
- `ticker(symbol)` -> `GET /ticker/{symbol}`
- `klines(params)` -> `GET /klines`
- `l2Book(params)` -> `GET /l2book`
- `stats(params)` -> `GET /stats`
- `riskSurfaces(market)` -> `GET /riskSurfaces?market=...`

## Account client

Implementation: `src/http/account_client.ts`.

Most account history methods call `POST /account` with a query type and then extract the expected payload from the
polymorphic response array. Decode failures throw `BulkDecodeError`.

```ts
const accountPublicKey = client.accountPublicKey ?? "BASE58_ACCOUNT_PUBLIC_KEY";

const full = await client.account.fullAccount(accountPublicKey);
const openOrders = await client.account.openOrders(accountPublicKey);
const fills = await client.account.fills(accountPublicKey);
const positions = await client.account.positions(accountPublicKey);
const funding = await client.account.fundingHistory(accountPublicKey);
const orderHistory = await client.account.orderHistory(accountPublicKey);
const feeTier = await client.account.feeTier(accountPublicKey);

const feeState = await client.account.feeState();
const proposals = await client.account.multisigProposals("BASE58_MULTISIG_PUBLIC_KEY");
```

Endpoint mapping:

- `fullAccount(user)` -> `POST /account`, type `fullAccount`
- `openOrders(user)` -> `POST /account`, type `openOrders`
- `fills(user)` -> `POST /account`, type `fills`
- `positions(user)` -> `POST /account`, type `positions`
- `fundingHistory(user)` -> `POST /account`, type `fundingHistory`
- `orderHistory(user)` -> `POST /account`, type `orderHistory`
- `feeTier(user)` -> `POST /account`, type `feeTier`
- `feeState()` -> `GET /feeState`
- `multisigProposals(pubkey)` -> `GET /multisig/{pubkey}/proposals`

## WebSocket client

Implementation: `src/ws/ws_client.ts`.

Responsibilities:

- connection lifecycle: `connect()`, `close()`
- subscriptions: `subscribe()` routed by topic through `WsRouter`
- request/response transaction submit: `post()` with `pendingPosts`

```ts
await client.ws.connect();

const handle = await client.ws.subscribe({ type: "ticker", symbol: "BTC-USD" }, (msg) => {
  console.log("ticker update", msg);
});

await handle.unsubscribe();
await client.ws.close();
```

## Trading pipeline

Main implementation: `src/http/trade_client.ts`.

Keep the trading path linear:

1. SDK params
2. builder output
3. signer
4. normalized signed transaction
5. submit over HTTP or WebSocket

Key files:

- Builders: `src/builders/orders.ts`
- Signing adapter: `src/signing/keychain_signer.ts`
- Normalization: `src/signing/normalize_signed_transaction.ts`
- Submission: `src/http/trade_client.ts`

`TradeClient.submit()` defaults to HTTP `POST /order`. Pass `{ via: "ws" }` to submit through `client.ws.post()`.

By default, a structured response with `status: "error"` throws `BulkTransactionRejectedError`. Disable that with
`{ throwOnReject: false }` when the caller needs to inspect rejected responses directly.

### Place orders and cancel

```ts
const limit = await client.trade.placeLimitOrder({
  symbol: "BTC-USD",
  side: "buy",
  price: 50000,
  size: 0.1,
  tif: "GTC",
});

const market = await client.trade.placeMarketOrder({
  symbol: "BTC-USD",
  side: "sell",
  size: 0.05,
});

await client.trade.cancelOrder({
  symbol: "BTC-USD",
  orderId: "ORDER_ID_STRING",
});

await client.trade.cancelAll({ symbols: ["BTC-USD", "ETH-USD"] });
```

Order IDs for resting limit orders are nested in response statuses:

```ts
const firstStatus = limit.response.data.statuses[0];
const orderId =
  firstStatus && "resting" in firstStatus && firstStatus.resting
    ? firstStatus.resting.oid
    : undefined;
```

### Batch

```ts
import { toKeychainCancelOrder, toKeychainLimitOrder } from "bulk-ts-sdk";

const res = await client.trade.batch([
  toKeychainLimitOrder({
    symbol: "BTC-USD",
    side: "buy",
    price: 50000,
    size: 0.1,
  }),
  toKeychainCancelOrder({
    symbol: "BTC-USD",
    orderId: "ORDER_ID_STRING",
  }),

```

### WebSocket submit

```ts
await client.ws.connect();

const res = await client.trade.placeLimitOrder(
  {
    symbol: "BTC-USD",
    side: "buy",
    price: 50000,
```

### Agent wallet management

```ts
const agentWalletClient = new BulkClient({
  privateKey: process.env.AGENT_WALLET_PRIVATE_KEY!,

const agentWalletPublicKey = agentWalletClient.accountPublicKey;

  agent: agentWalletPublicKey,
  remove: true,

```

## Signer details

Implementation: `src/signing/keychain_signer.ts`.

- `KeychainSigner.fromPrivateKey(privateKey, targetAccountPublicKey?)`
- `accountPublicKey` getter returns `targetAccountPublicKey ?? nativeSigner.pubkey`
- `sign(input)` accepts `KeychainOrderInput | AgentWalletKeychainInput`
- `signGroup(inputs)` and `signAll(inputs)` accept order/cancel inputs only
- `nextNonce()` is monotonic so rapid signing cannot reuse a nonce

Do not mutate signed transactions after native signing to force a different target account; that produces invalid
signatures.

## Errors

Defined in `src/errors.ts`:

- `BulkHttpError`: non-2xx HTTP response, includes `status` and parsed data.
- `BulkWsError`: WebSocket protocol or connection failure.
- `BulkTimeoutError`: HTTP or WebSocket timeout.
- `BulkDecodeError`: response did not contain the expected payload shape.
- `BulkTransactionRejectedError`: trading endpoint returned a structured reject.

```ts
import {
  BulkDecodeError,
  BulkHttpError,
  BulkTimeoutError,
  BulkTransactionRejectedError,
  BulkWsError,
} from "bulk-ts-sdk";

try {
  await client.trade.placeLimitOrder({
    symbol: "BTC-USD",
    side: "buy",
    price: 50000,
    size: 0.1,
  });
} catch (err) {
  if (err instanceof BulkTransactionRejectedError) {
    console.error("rejected", err.response);
  } else if (err instanceof BulkHttpError) {
    console.error("http", err.status, err.data);
  } else if (err instanceof BulkWsError) {
    console.error("ws", err.message);
  } else if (err instanceof BulkTimeoutError) {
    console.error("timeout", err.message);
  } else if (err instanceof BulkDecodeError) {
    console.error("decode", err.message);
  } else {
    throw err;
  }
}
```

## E2E environment variables

E2E tests read `.env`.

- `MAIN_WALLET_PRIVATE_KEY`: main wallet private key. Required by itself for standard account/trade E2E.
- `AGENT_WALLET_PRIVATE_KEY`: agent wallet private key. Use with `MAIN_WALLET_PUBLIC_KEY` for agent-wallet E2E.
- `MAIN_WALLET_PUBLIC_KEY`: main wallet public key. Required with `AGENT_WALLET_PRIVATE_KEY` when an agent wallet
  targets the main wallet.
- `MULTISIG_PUBKEY`: optional multisig account public key. Enables multisig proposal E2E.
- `BULK_HTTP_URL`: optional HTTP API override.
- `BULK_WS_URL`: optional WebSocket API override.

`MAIN_WALLET_PRIVATE_KEY` is the simple main-wallet mode. `AGENT_WALLET_PRIVATE_KEY` plus `MAIN_WALLET_PUBLIC_KEY` is
the agent-wallet mode. Do not collapse the private key variables.

## Extending the SDK

When adding or changing endpoints:

- Put public market data in `MarketClient`.
- Put account and multisig reads in `AccountClient`.
- Put signing/submission flows in `TradeClient`.
- Put realtime subscriptions and WS submit behavior in `WsClient`.
- Add or update E2E coverage for the public SDK method.
- Keep builders, `KeychainOrderInput`, `AgentWalletKeychainInput`, signer behavior, and normalization aligned for trade
  changes.
